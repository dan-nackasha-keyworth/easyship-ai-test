Model decides for itself which read-only lookup tools (if any) to call on low-confidence messages, bounded to 4 iterations. Everything else in the build is a fixed-order workflow; this is the exception. Ran on 41/100 messages in the latest run.

Entry-channel modeling Support/Sales as a prior, not a rule
Done
7 deliberate override cases authored across the set. Result: 6/7 correctly classified by content despite the "wrong" channel; the 7th correctly escalated to Team Lead Triage rather than guessed.
✓
Routing fix + reference-grounded drafting + Enterprise AE Major routing correction, Help Centre/playbook grounding, second confidence score
Done
Four changes from one round of design review, tested together: (1) contradictory-signals messages now route to Team Lead Triage instead of defaulting to Success, so Success stops absorbing ambiguous technical escalations; (2) formal Close Account/Cancel Shipment requests stay Support-owned, looping in Success only for large accounts (new deterministic arr_band lookup, regex-based formal-request detection - caught and fixed a real gap in testing where "close our account" didn't match an earlier exact-phrase list); (3) a Sales message's stated shipment volume in the top band(s) now sets a distinct "Enterprise AE" handling path, mirroring the real Sales form's volume field; (4) drafting is now grounded in a matched mock Help Centre/playbook article when one exists (find_matching_article, keyword overlap, no extra API call), feeding a new second confidence score - score_draft_confidence - that answers "is this draft good enough to send," distinct from routing confidence. The agentic investigation step also gained a third tool, search_help_centre. Verified via 5 synthetic unit-test scenarios, a 4-message real-API test batch (results/routing_fix_check_run.html), and a fresh full 100-message dev run.
